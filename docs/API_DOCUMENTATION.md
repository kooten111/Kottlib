# Kottlib API Documentation (YACReader Protocol Compatible)

This document details the API endpoints for Kottlib, covering both v1 and v2 versions.

## Overview

The server uses a custom HTTP server implementation. Requests are routed based on regex patterns defined in `requestmapper.cpp`.
- **v1 API**: Primarily uses custom text-based formats (key-value pairs or delimited strings) and HTML for the web UI.
- **v2 API**: Introduces JSON for most metadata endpoints, but retains some custom text formats for specific operations (like comic opening).

### Data Formats (v1)

#### Folder Info (`/library/{id}/folder/{folderId}/info`)
Returns a list of comics and subfolders in a custom delimited format.
**Content-Type**: `text/plain; charset=utf-8`

**Format**:
```text
/library/{libId}/comic/{comicId}:{fileName}:{fileSize}
...
```
*Note: Subfolders are recursively serviced, but the output format for folders is not explicitly distinguished in the controller code examined, suggesting it might just list comics recursively or use a similar format.*

#### Comic Download Info (`/library/{id}/comic/{comicId}`)
**Content-Type**: `text/plain; charset=utf-8`

**Format**:
```text
fileName:{fileName}
fileSize:{fileSize}
```

#### Comic Info / Remote (`/library/{id}/comic/{comicId}/info` or `/remote`)
**Content-Type**: `text/plain; charset=utf-8`

**Format**:
```text
library:{libraryName}
libraryId:{libraryId}
previousComic:{prevId} (only if remote)
nextComic:{nextId} (only if remote)
{key}:{value}
...
```
The `{key}:{value}` pairs are generated by `ComicDB::toTXT()`.

### Enhanced Data Formats

#### Series List (`/v2/library/{id}/series`)
**Content-Type**: `application/json`

Returns an array of series with aggregated metadata and reading progress:

```json
[
  {
    "id": 123,
    "name": "Series Name",
    "title": "Series Name",
    "series": "Series Name",
    "series_name": "Series Name",
    "volumes": [
      {
        "id": 456,
        "title": "Volume 1",
        "volume": 1,
        "issue_number": 1,
        "filename": "series_v01.cbz",
        "hash": "abc123...",
        "num_pages": 200,
        "current_page": 0,
        "is_completed": false,
        "progress_percent": 0
      }
    ],
    "publisher": "Publisher Name",
    "total_issues": 10,
    "cover_hash": "abc123...",
    "first_comic_id": 456,
    "is_standalone": false,
    "year": 2023,
    "writer": "Writer Name",
    "artist": "Artist Name",
    "genre": "Action",
    "synopsis": "Series description..."
  }
]
```

**Query Parameters:**
- `sort`: Sort order - `name`, `recent`, `updated`, `progress` (default: `name`).
- `recent` sorts by series/comic creation time; `updated` sorts by latest newly-added content in a series subtree.
- Alias inputs are normalized: `date_added`/`recently_added` -> `recent`, `last_updated`/`recently_updated`/`date_updated` -> `updated`.
- `include_metadata`: Include series metadata (default: `true`)

#### Series Tree (`/v2/libraries/series-tree`)
**Content-Type**: `application/json`

Returns hierarchical tree structure with caching for performance:

```json
[
  {
    "id": 1,
    "name": "Library Name",
    "type": "library",
    "children": [
      {
        "id": 10,
        "name": "Folder Name",
        "type": "folder",
        "libraryId": 1,
        "comicCount": 50,
        "children": [...]
      }
    ]
  }
]
```

### Key Features of Extensions

1. **Series Detection**: Automatic series grouping and metadata aggregation
2. **Advanced Search**: Full-text search with POST support for complex queries
3. **Reading Progress Sync**: Enhanced progress tracking across all endpoints
4. **Collection Management**: Full CRUD operations for favorites, tags, and reading lists
5. **Performance Optimization**: Cached series trees and optimized SQL queries
6. **Metadata Enrichment**: Scanner integration for automatic metadata enhancement

### Compatibility Notes

- All YACReader v1 and v2 endpoints remain **fully functional**
- Mobile apps using standard YACReader API will work without modification
- Enhanced endpoints are **optional** - apps can progressively adopt them
- Response formats maintain YACReader conventions (string IDs, field names, etc.)
